# 12_문항 랜덤화 및 결과 표시 정책

## 1. 목적

본 문서는 2026-06-22 기준 구현된 문항 후보 랜덤화와 결과 축 게이지 표시 정책을 정리합니다. 목표는 사용자가 같은 테스트를 반복해도 너무 비슷한 문항만 보지 않게 하면서, 기존 20문항 알고리즘과 결과 산출의 재현성을 유지하는 것입니다.

## 2. 문항 랜덤화 원칙

문항 랜덤화는 전체 문항을 무작위로 섞지 않습니다. 서버가 먼저 알고리즘 슬롯을 결정하고, 그 슬롯 안에 있는 후보 문항 중 하나를 `questionSeed` 기반으로 선택합니다.

슬롯 기준:

| 단계 | 슬롯 키 |
|---|---|
| Phase 1 | `axis + commonOrder` |
| Phase 2 단방향 | `axis + branchCondition + branchOrder` |
| Phase 2 BALANCED | `axis + BALANCED + branchOrder` |

현재 구현:

| 구분 | 기본 슬롯 | V2 후보 | V3 후보 | 서버 후보 합계 |
|---|---:|---:|---:|---:|
| Phase 1 | 8 | 8 | 8 | 24 |
| Phase 2 | 36 | 36 | 36 | 108 |
| 합계 | 44 | 44 | 44 | 132 |

즉, 현재 MVP는 44개 알고리즘 슬롯과 132개 후보 문항을 기준으로 합니다.

## 3. Seed와 복구

- 클라이언트는 테스트 세션 시작 시 `poldigm.questionSeed`를 생성해 `sessionStorage`에 저장합니다.
- `/api/next-question` 요청에는 `history`와 `questionSeed`가 함께 전달됩니다.
- 서버는 먼저 같은 세션에서 덜 사용된 `copyFamily` 후보를 우선하고, 최근 3문항에서 반복된 `scenarioTag`, `toneTag` 후보를 뒤로 미룬 뒤, 전체 사용량과 `questionSeed + slotId + current` 조합으로 후보 문항을 deterministic하게 선택합니다.
- 새로고침 후에도 같은 seed와 history가 남아 있으므로 같은 진행률에서 같은 문항이 복구됩니다.

이 방식은 사용자에게는 랜덤하게 보이지만, QA와 결과 산출에는 재현 가능한 흐름을 제공합니다.

## 4. 서버 전용 유지 항목

클라이언트에는 선택된 공개 문항만 내려갑니다. 다음 항목은 계속 서버 전용입니다.

- 전체 후보 문항 풀
- `scoreEffect`
- `branchCondition`
- `intensityEffect`
- `metadata.scenarioTag`
- `metadata.copyFamily`
- `metadata.toneTag`
- 슬롯별 후보 목록

공개 응답에는 `prompt`, `choices`, `display`, `progress`만 포함합니다.

## 5. 후보 문항 추가 규칙

후보 문항을 추가할 때는 기존 슬롯의 variant로 추가합니다. 새 후보는 기존 문항과 같은 알고리즘 계약을 공유해야 합니다.

유지해야 하는 값:

- `axis`
- `phase`
- `branchCondition`
- `commonOrder` 또는 `branchOrder`
- `scoreEffect`
- `intensityEffect`

바꿀 수 있는 값:

- `prompt`
- `choices`
- `display.promptLines`
- `display.promptHighlights`
- `display.choices`

### 5.1. 권장 확장 단위

20문항 테스트에서 사용자에게 비슷한 문항이 반복 노출되지 않게 하려면, 노출 문항 수보다 큰 후보 풀이 필요합니다. 다만 알고리즘 슬롯을 무작정 늘리면 20문항 흐름과 16유형 도달성 검증이 흔들릴 수 있으므로, 우선은 기존 44개 슬롯에 variant를 추가합니다.

권장 단계:

| 단계 | 후보 구성 | 총 후보 수 | 용도 |
|---|---:|---:|---|
| MVP 최소 | 슬롯당 2개 후보 | 88 | 최소 구현 기준 |
| 운영 권장 | 슬롯당 3개 후보 | 132 | 현재 구현 기준, 반복 테스트 피로도 완화 |
| 장기 확장 | 핵심 슬롯 4~5개 후보 | 176~220 | A/B 카피 실험 및 응답 편향 조정 |

새 슬롯은 측정 축, 분기 수, 노출 문항 수를 바꿔야 할 때만 추가합니다. 단순히 문장을 다양화하거나 비슷한 딜레마를 다른 생활 장면으로 바꾸는 작업은 variant로 처리합니다.

문항 추가 방향:

| 추가 단위 | 권장 방식 | 피해야 할 방식 |
|---|---|---|
| 축 | `C-I`, `T-P`, `M-E`, `O-L` 네 축이 비슷한 밀도로 보강되게 추가 | 특정 축만 후보가 과도하게 많아지는 구성 |
| 상황 | 일상, 직장/학교, 온라인, 공공 규칙, 제도 판단처럼 서로 다른 장면으로 변형 | 같은 질문을 단어만 바꿔 반복 |
| 판단 구조 | 공동 기준과 개인 보호, 원칙 유지와 예외 인정, 성과 기준과 배려 비용처럼 충돌 지점이 보이게 작성 | 정답처럼 보이는 선택지를 한쪽에만 배치 |
| 표현 | 2줄 문항, 짧은 선택지 라벨, 한 문장 설명, 핵심어 강조를 모두 작성 | 긴 설명문이나 설문지식 찬반 문장으로 회귀 |

Phase 1의 8개 슬롯은 전체 축의 초기 기울기를 잡는 기준점입니다. 따라서 Phase 1 슬롯 자체를 먼저 늘리기보다, Phase 2 후보를 축별로 고르게 보강하고 운영 데이터에서 특정 축의 변별력이 낮을 때만 새 슬롯을 검토합니다.

### 5.2. 랜덤 노출 기준

문항 노출은 “완전 랜덤”이 아니라 “재현 가능한 랜덤”이어야 합니다.

좋은 방식:

- 서버가 먼저 20문항 알고리즘 슬롯을 결정합니다.
- 각 슬롯 안에서 `questionSeed`를 이용해 후보 문항 하나를 선택합니다.
- 같은 세션의 같은 진행률에서는 새로고침 후에도 같은 문항이 복구됩니다.
- 서버 검증은 문항 ID의 완전 일치가 아니라 같은 슬롯의 도달 가능한 후보인지 확인합니다.

피해야 할 방식:

- 클라이언트에서 전체 문항을 임의로 섞는 방식
- `Math.random()`만으로 서버 재현성이 없는 문항을 선택하는 방식
- 이미 답한 알고리즘 슬롯과 같은 슬롯을 다시 노출하는 방식

현재 MVP는 결과 재현성과 QA 편의성을 위해 20문항의 알고리즘 슬롯 순서를 고정합니다. 사용자가 체감하는 다양성은 같은 슬롯 안의 variant 선택, `scenarioTag`, `copyFamily`, `toneTag` 보정으로 만듭니다.

향후 슬롯 순서까지 랜덤화할 경우에는 아래 제약을 함께 둡니다.

- Phase 1 8문항은 네 축이 모두 2회씩 노출되어야 합니다.
- Phase 2 12문항은 네 축이 모두 3회씩 노출되어야 합니다.
- 같은 `axis`가 3회 이상 연속 노출되지 않아야 합니다.
- 같은 `scenarioTag` 또는 같은 `toneTag`가 가까운 순번에 과도하게 반복되지 않아야 합니다.
- 동일한 `questionSeed`와 동일한 `history`에서는 항상 같은 다음 문항이 나와야 합니다.
- 서버 검증은 랜덤 순서에서도 도달 가능한 슬롯과 후보 문항인지 확인해야 합니다.

### 5.3. 비슷한 문항 반복 방지 확장

운영 단계에서 후보 문항이 슬롯당 3개 이상으로 늘어나면, 단순히 seed만 사용하는 것보다 문항 메타를 함께 두는 편이 좋습니다. 현재 구현은 모든 후보 문항에 아래 서버 전용 메타를 이미 보유합니다.

| 메타 | 예시 | 용도 |
|---|---|---|
| `scenarioTag` | `privacy`, `education`, `workplace-reward`, `public-space`, `principle-pressure` | 같은 생활 소재나 추상 판단 프레임이 너무 가까운 순번에 반복되지 않도록 조정 |
| `copyFamily` | `base-copy`, `variant-v2`, `variant-v3` | 같은 세션에서 특정 표현 계열이 과도하게 반복되지 않도록 보정 |
| `toneTag` | `daily`, `institutional`, `crisis` | 모든 문항이 위기 상황처럼 읽히는 문제를 완화 |

권장 우선순위:

1. MVP와 1차 운영에서는 현재처럼 슬롯 고정 + seed 기반 variant 선택을 유지합니다.
2. 현재처럼 슬롯당 3개 후보, 총 132개 후보를 유지하면서 `scenarioTag`, `copyFamily`, `toneTag` 값을 기준으로 후보를 묶습니다.
3. 같은 테스트 세션 안에서 특정 `copyFamily`가 많이 사용된 경우, 같은 슬롯의 다른 후보를 우선 선택하는 방식으로 보정합니다.
4. 최근 3문항 안에서 이미 사용된 `scenarioTag`, `toneTag`와 같은 후보는 뒤로 미뤄, 바로 이어지는 문항들이 비슷한 생활 소재나 긴장도로 읽히지 않게 합니다.
5. 그 다음 전체 세션의 `scenarioTag`, `toneTag` 사용량을 보정합니다.
6. `questionSeed`는 동률 후보의 결정 순서를 정하는 데 사용하므로, 새로고침 복구와 표현 계열 분산을 함께 만족해야 합니다.

이 확장은 문항의 알고리즘 슬롯을 바꾸지 않습니다. `axis`, `phase`, `branchCondition`, `scoreEffect`, `intensityEffect`는 그대로 유지하고, 후보 선택의 체감 다양성을 높이는 보조 정보로만 사용합니다.

## 6. 결과 퍼센트 표시 정책

서버가 반환하는 `scores`는 원점수입니다. 결과 코드와 DB 저장 근거는 이 원점수를 사용합니다.

하지만 결과 화면의 축 게이지는 원점수를 그대로 퍼센트로 보여주지 않습니다. 한쪽 선택이 강한 사용자에게 `100%`처럼 단정적인 숫자를 보여주면, Poldigm이 의도한 “사회적·정치적 가치관의 경향”보다 확정적 진단처럼 읽힐 수 있기 때문입니다.

현재 서버는 `/api/submit-result` 응답의 `resultViewModel.axisGauges`에 표시용 퍼센트를 포함합니다. UI는 이 값을 렌더링하며, `scores` 원점수를 다시 퍼센트로 해석하지 않습니다.

- 원점수 50%는 50%로 표시
- 한쪽 극단값은 최대 88% 수준으로 완화
- 보조 문구는 `강하게 기울어짐`, `기울어짐`, `균형에 가까움`을 사용

따라서 `scores`와 `resultViewModel.axisGauges`의 퍼센트는 의도적으로 다를 수 있습니다.

표시 정책:

| 원점수 상태 | 표시 방식 |
|---|---|
| `100 : 0` 같은 극단값 | `88 : 12` 수준으로 완화 |
| `80 : 20` 이상 | 강한 경향으로 표시 |
| `60 : 40` 안팎 | 완만한 경향으로 표시 |
| `50 : 50` | 균형에 가까움으로 표시 |

이 보정은 사용자의 성향을 흐리기 위한 것이 아니라, Poldigm이 “확정적 진단”이 아니라 “사회적·정치적 가치관의 경향”을 보여주는 서비스라는 점을 유지하기 위한 UI 정책입니다. 결과 코드, DB 저장, QA 검증은 계속 원점수 기준으로 수행합니다.

예시:

| 원점수 | 표시용 게이지 | 사용자 해석 |
|---|---|---|
| `C 100 : I 0` | `C 88 : I 12` | 공동체 쪽으로 강하게 기울어짐 |
| `T 60 : P 40` | `T 58 : P 42` | 전통 쪽으로 약하게 기울어짐 |
| `M 50 : E 50` | `M 50 : E 50` | 균형에 가까움 |

결과 화면에서는 숫자만 단독으로 강조하지 않고, 각 축의 양쪽 표시용 퍼센트와 방향 라벨, 강도 라벨을 함께 보여줍니다. 예를 들어 `C 88 : I 12`처럼 왼쪽/오른쪽 값을 나란히 보여주고, 보조 문구로 `C 공동체 쪽 · 강하게 기울어짐`을 붙입니다. 목표는 “100% 어떤 사람”처럼 단정하는 것이 아니라, 각 축에서 어느 쪽으로 얼마나 기울어져 있는지 이해하게 만드는 것입니다.

## 8. 운영 의사결정 요약

문항 확장, 랜덤 노출, 결과 퍼센트 표시는 아래 기준을 우선합니다.

| 질문 | 권장 판단 | 이유 |
|---|---|---|
| 문항을 더 추가하려면 어떻게 추가해야 하는가? | 20문항 노출 수를 늘리기보다 기존 44개 알고리즘 슬롯 안의 variant를 축별로 고르게 추가 | 검사 길이를 늘리지 않고도 반복 테스트 피로도를 낮추며, 기존 16유형 도달성과 점수 체계를 유지할 수 있음 |
| 사용자에게 비슷한 문항이 계속 보이지 않게 하려면? | 완전 랜덤이 아니라 `questionSeed` 기반 슬롯 내 후보 선택과 `copyFamily`, `toneTag`, `scenarioTag` 보정을 사용 | 사용자에게는 다양하게 보이지만, 같은 세션의 새로고침 복구와 QA 재현성을 함께 보장할 수 있음 |
| 결과의 각 항목 퍼센트가 모두 100%처럼 보이면? | 원점수 `scores`는 유지하고, 결과 화면은 `resultViewModel.axisGauges`의 완화된 표시용 퍼센트만 사용 | 사회적·정치적 가치관을 확정 판정처럼 보이게 하지 않고, 축별 기울기와 강도를 더 자연스럽게 전달할 수 있음 |

운영 문항 풀을 확장할 때의 1차 목표는 “더 많은 문항을 묻기”가 아니라 “같은 20문항 흐름 안에서 덜 반복적으로 느껴지는 문항 경험을 만들기”입니다. 따라서 Phase 1의 8개 초기 기준 슬롯은 안정적으로 유지하고, Phase 2 후보를 축별·분기별로 먼저 보강합니다.

결과 화면의 숫자는 정확도나 정답률이 아닙니다. `C 88 : I 12` 같은 표시는 공동체 쪽 응답 경향이 강하다는 UI 표현이며, 내부 계산에 쓰이는 원점수 `C 100 : I 0`을 그대로 노출하지 않기 위한 완화 정책입니다.
