# 📄 03_IA_정보구조_설계서

## 1. 문서 개요

본 문서는 폴다임(Poldigm)의 정보구조(IA, Information Architecture)를 정의합니다. PRD의 기능 요구사항과 UI/UX 화면설계서를 연결하여, 사용자가 어떤 경로로 진입하고 어떤 화면을 거쳐 결과를 확인하는지, 각 화면이 어떤 데이터를 필요로 하는지 명확히 합니다.

로컬 MVP 구현 시에는 본 문서의 라우트와 화면 구조를 기준으로 구현하되, 저장/배포/도메인 관련 정책은 `33_로컬_MVP_구현_계획서`를 따릅니다.

## 2. IA 설계 원칙

- 사용자는 로그인, 회원가입, 권한 요청 없이 테스트를 시작할 수 있어야 한다.
- 핵심 플로우는 `랜딩 → 테스트 → 분석 → 결과 → 공유/재시작`으로 단순화한다.
- 테스트 진행 중에는 문항 선택 외의 행동을 최소화한다.
- 서버 전용 문항 데이터, 점수 배점, 분기 조건은 클라이언트 정보구조에 노출하지 않는다.
- 결과 화면은 공유와 재참여를 유도하는 최종 도착점으로 설계한다.
- 법무/정책 화면은 핵심 플로우를 방해하지 않되, 푸터 또는 보조 링크로 접근 가능해야 한다.

## 3. 사이트맵

```text
/
├─ /test
├─ /analysis
├─ /result/[id]
├─ /terms
├─ /privacy
└─ +error.svelte
```

로컬 MVP에서 `/terms`, `/privacy`는 최소 정적 페이지로 구현합니다. 잘못된 라우트와 예외 화면은 SvelteKit 전역 오류 화면인 `src/routes/+error.svelte`에서 처리합니다.

`/analysis`는 IA상 권장 라우트지만 현재 로컬 MVP에서는 별도 라우트를 만들지 않고 `/test` 내부 로딩 상태로 대체합니다. 별도 라우트로 분리하는 시점은 실제 MVP에서 분석 이벤트, SEO/robots 메타, 화면 전환 추적이 필요해질 때로 봅니다.

## 4. 페이지별 정보구조

| 라우트 | 화면명 | 목적 | 주요 콘텐츠/기능 | 필수 여부 |
|---|---|---|---|---|
| `/` | 랜딩 | 테스트 시작 유도 | 로고, 메인 카피, 서브 카피, 참여자 수, 시작 CTA | 필수 |
| `/test` | 테스트 진행 | 20문항 응답 수집 | 진행률, 단계명, 질문, A/B 선택지, 뒤로가기 | 필수 |
| `/analysis` | 분석/로딩 | 결과 계산 대기와 기대감 형성 | 분석 문구, 키워드 애니메이션, 진행률 | 권장 |
| `/result/[id]` | 결과 | 결과 해석과 공유 유도 | 유형명, 캐릭터, 4축 점수, 설명, 궁합, 공유/저장/재시작 | 필수 |
| `/terms` | 이용약관 | 법적 고지 | 서비스 이용 조건 | 로컬 MVP 구현 완료, 실제 MVP 전 운영 주체 갱신 |
| `/privacy` | 개인정보처리방침 | 개인정보/쿠키/통계 고지 | 수집 항목, 보관 정책, 문의처 | 로컬 MVP 구현 완료, 실제 MVP 전 운영 주체 갱신 |
| `+error.svelte` | 오류 | 잘못된 접근 처리 | 홈 이동 CTA, `noindex,nofollow` | 구현 완료 |

## 5. 핵심 유저 플로우

### 5.1. 신규 테스트 플로우

```text
SNS/커뮤니티 링크
→ 랜딩 페이지
→ 테스트 시작
→ 20문항 응답
→ 분석 화면
→ 결과 화면
→ 이미지 저장 또는 링크 복사
→ 테스트 다시 하기 또는 이탈
```

### 5.2. 테스트 진행 중 새로고침 플로우

```text
/test 진입
→ sessionStorage에 응답 이력 존재 확인
→ 응답 이력이 AnswerHistoryItem 최소 shape를 통과하면 현재 진행 상태 복구
→ 이력이 없으면 첫 문항부터 시작
```

### 5.3. 결과 재진입 플로우

```text
/result/[id] 진입
→ 결과 데이터 조회
→ 조회 성공 시 결과 렌더링
→ 로컬 MVP에서 조회 실패 시 sessionStorage fallback 확인
→ fallback의 resultId, PublicResult 최소 shape, 공식 8개 점수 키와 0~100 정수 점수값, scores와 axisGauges 계산 일치, 궁합 유형 코드, 캐릭터 파일명 계약이 맞으면 결과 렌더링
→ fallback도 없으면 테스트 다시 시작 CTA 표시
```

실제 MVP에서 결과 URL 공유/재방문을 지원하려면 Supabase에 저장된 결과 ID 조회를 기준으로 복구합니다.

## 6. 화면별 데이터 요구사항

### 6.1. 랜딩 `/`

필요 데이터:

- 서비스명 `Poldigm`
- 메인 카피
- 서브 카피
- 참여자 수 또는 mock 참여자 수
- 시작 CTA 라벨

사용자 액션:

- `테스트 시작하기`

상태 처리:

- 시작 시 기존 테스트 진행 상태를 초기화한다.
- 로컬 MVP에서는 참여자 수를 고정값 또는 mock 값으로 표시한다.

### 6.2. 테스트 `/test`

필요 데이터:

- 현재 문항 번호
- 전체 문항 수 `20`
- Phase 표시
- 문항 텍스트
- A/B 선택지 텍스트
- `questionId`

사용자 액션:

- A 선택
- B 선택
- 뒤로가기

상태 처리:

- 선택 즉시 `sessionStorage`에 응답 이력을 저장한다.
- 다음 문항은 `/api/next-question` 응답으로 갱신한다.
- 20번째 응답 후 `/api/submit-result`를 호출한다.
- API 오류 시 재시도 또는 테스트 재시작 CTA를 제공한다.

클라이언트에 노출하면 안 되는 정보:

- `scoreEffect`
- `branchCondition`
- `intensityEffect`
- 전체 문항 풀

### 6.3. 분석 `/analysis`

필요 데이터:

- 분석 진행 문구
- 선택 키워드 또는 고정 키워드
- 진행률 표시 값

사용자 액션:

- 없음

상태 처리:

- `/api/submit-result` 완료 후 결과 ID를 받아 `/result/[id]`로 이동한다.
- 현재 로컬 MVP에서는 구현 단순화를 위해 `/test` 내부의 임시 로딩 상태로 대체한다.

### 6.4. 결과 `/result/[id]`

필요 데이터:

- `resultId`
- 결과 코드 예: `CTMO-S`
- 유형명
- 한 줄 요약
- 상세 설명
- 4축 점수
- S/M 강도 태그
- 캐릭터 이미지 또는 placeholder
- 베스트 궁합
- 워스트 궁합
- 공유 URL

사용자 액션:

- 이미지 저장
- 링크 복사
- 테스트 다시 하기
- 선택적으로 인스타그램 스토리 공유

상태 처리:

- 결과 조회 실패 시 같은 세션의 `sessionStorage` fallback을 확인한다.
- fallback은 URL의 `resultId`와 일치하고 공개 결과 데이터의 최소 shape, 공식 8개 점수 키와 0~100 정수 점수값, `scores`에서 재계산한 `axisGauges`, `winnerPercent`, 방향 라벨, 강도 라벨, `chemistryBest`/`chemistryWorst` 유형 코드, `character_{typeCode}.png` 파일명 계약이 모두 일치할 때만 사용한다.
- fallback도 없으면 결과 내용을 임의 생성하지 않고 테스트 다시 시작 CTA를 표시한다.
- 다시 하기 선택 시 테스트 진행 상태와 결과 fallback을 초기화한다.

## 7. 내비게이션 구조

폴다임은 테스트 몰입도를 위해 일반적인 다단 메뉴를 두지 않습니다.

| 위치 | 요소 | 동작 |
|---|---|---|
| 랜딩 상단 | 로고 | `/` 이동 |
| 랜딩 하단 | 테스트 시작 CTA | `/test` 이동 및 상태 초기화 |
| 테스트 상단 | 뒤로가기 | 이전 문항으로 이동. 첫 문항에서는 `/` 이동 또는 확인 후 종료 |
| 결과 하단 | 이미지 저장 | 결과 캡처 영역 저장 |
| 결과 하단 | 링크 복사 | 현재 결과 URL 복사 |
| 결과 하단 | 테스트 다시 하기 | 상태 초기화 후 `/test` 이동 |
| 푸터 | 이용약관 | `/terms` 이동 |
| 푸터 | 개인정보처리방침 | `/privacy` 이동 |

## 8. 상태와 저장 구조

### 8.1. 클라이언트 상태

| 상태 | 저장 위치 | 용도 | 만료 정책 |
|---|---|---|---|
| 응답 이력 | `sessionStorage` | 새로고침 복구, 다음 문항 요청 | 탭 종료 시 소멸 |
| 현재 진행 상태 | `sessionStorage` | `/test` 복구 | 탭 종료 시 소멸 |
| 마지막 결과 fallback | `sessionStorage` | 로컬 결과 페이지 복구 | 탭 종료 시 소멸 |
| 공유 URL | 런타임 계산 | 링크 복사 | 저장하지 않음 |

### 8.2. 서버 상태

| 상태 | 로컬 MVP | 실제 MVP |
|---|---|---|
| 문항 풀 | 서버 전용 `questions.ko-KR.ts` 모듈 | 서버 전용 로케일별 TS 모듈 |
| 점수/분기 로직 | 서버 전용 모듈 | 서버 전용 모듈 |
| 결과 저장 | 메모리 mock 저장소 | Supabase `test_results` |
| 봇 검증 | SvelteKit dev 런타임 local mock token | Cloudflare Turnstile |

## 9. 오류 및 예외 화면

| 상황 | 사용자 메시지 방향 | CTA |
|---|---|---|
| `/api/next-question` 실패 | 다음 문항을 불러오지 못함 | 다시 시도, 처음부터 시작 |
| 응답 이력 검증 실패 | 진행 정보가 올바르지 않음 | 테스트 다시 시작 |
| `/api/submit-result` 실패 | 결과를 계산하지 못함 | 다시 시도, 처음부터 시작 |
| 결과 ID 없음 | 결과를 찾을 수 없음 | 테스트 다시 시작 |
| 잘못된 라우트 | 페이지를 찾을 수 없음 | 홈으로 이동 |

오류 화면은 긴 설명보다 복구 행동을 우선합니다.

## 10. 로컬 MVP 우선순위

로컬 MVP에서 반드시 구현할 IA 범위:

- `/`
- `/test`
- `/result/[id]`
- 테스트 진행 중 `sessionStorage` 복구
- 결과 페이지 `sessionStorage` fallback
- 테스트 다시 하기
- `/terms`, `/privacy` 최소 정적 고지 페이지
- 링크 복사 또는 이미지 저장 중 최소 1개
- 잘못된 라우트 복구용 전역 오류 화면

로컬 MVP에서 선택 구현 가능한 범위:

- `/analysis` 별도 라우트
- 실제 참여자 수
- 실제 SNS 공유 API

## 12. 현재 구현 반영

2026-06-22 기준 로컬 MVP 구현은 다음 IA를 따릅니다.

| 라우트 | 구현 파일 | 상태 |
|---|---|---|
| `/` | `src/routes/+page.svelte` | 구현 완료 |
| `/test` | `src/routes/test/+page.svelte` | 구현 완료. `/analysis`는 내부 로딩 상태로 대체 |
| `/result/[id]` | `src/routes/result/[id]/+page.server.ts`, `+page.svelte` | 구현 완료. mock 저장소 + sessionStorage fallback |
| `/terms` | `src/routes/terms/+page.svelte` | 구현 완료. 로컬 MVP 기준 최소 약관 고지 |
| `/privacy` | `src/routes/privacy/+page.svelte` | 구현 완료. 로컬 MVP 기준 최소 개인정보 고지 |
| 전역 오류 화면 | `src/routes/+error.svelte` | 구현 완료. 잘못된 라우트에 홈 이동 CTA와 `noindex,nofollow` 제공 |
| `/api/next-question` | `src/routes/api/next-question/+server.ts` | 구현 완료 |
| `/api/submit-result` | `src/routes/api/submit-result/+server.ts` | 구현 완료 |

로컬 MVP에서는 `/terms`, `/privacy`를 최소 정적 페이지로 제공합니다. `/analysis`는 별도 라우트로 만들지 않고 `/test` 내부 로딩 상태로 대체하며, 잘못된 라우트는 전역 오류 화면에서 비색인 처리합니다.
