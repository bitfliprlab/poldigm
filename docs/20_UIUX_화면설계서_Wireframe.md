# 📄 20_UIUX_화면설계서_Wireframe

## 1. 문서 개요

본 문서는 폴다임(Poldigm) 서비스의 모바일 웹 화면 설계도(Wireframe)입니다. 앱 설치 없이 SNS 인앱 브라우저에서 즉시 실행되는 특성을 고려하여 **모바일 반응형(Mobile-first)** 레이아웃을 기준으로 작성되었습니다. 디자이너와 프론트엔드 개발자는 본 명세의 화면 배치와 인터랙션 요소를 바탕으로 UI를 구현합니다.

## 2. 공통 UI/UX 원칙 (Global Rules)

- **Zero Friction (제로 마찰력):** 화면 전환 시 불필요한 팝업, 로그인, 권한 요구를 철저히 배제합니다.
    
- **One Screen, One Action:** 한 화면에는 오직 하나의 뚜렷한 행동(버튼 클릭)만 요구하여 인지 부하를 최소화합니다.
    
- **Micro-Interaction:** 선택지를 누를 때 선택 카드의 눌림/포커스 상태와 진행률 변화를 짧게 보여줍니다. 과한 스와이프/게임식 연출보다 빠른 응답감을 우선합니다.
    

## 3. 화면별 레이아웃 및 기능 명세

### 3.1. 메인 랜딩 화면 (Landing Page)

유저가 링크를 클릭하고 최초로 도달하는 화면입니다. 3초 안에 호기심을 자극하고 이탈을 막아야 합니다.

- **[Top] 헤더 영역**
    
    - **Logo:** 좌측 상단 `Poldigm` 텍스트 로고 (미니멀하고 세련된 산세리프 폰트).
        
- **[Center] 메인 카피**
    
    - **Label:** "정답 없는 선택지"
        
    - **Main Copy:** "가치관 밸런스 체크"
        
    - **Sub Copy:** "20개 질문에 답하면 내 선택 기준을 가볍게 정리해줘요."

    - **Meta Chips:** `20문항`, `로그인 없음`, `약 3분` 같은 보조 정보를 작은 pill 형태로 배치합니다. 랜딩 첫 화면에는 축 카드나 대형 통계 카드를 두지 않습니다.
        
- **[Bottom] 액션 영역 (Floating CTA)**
    
    - **Social Proof (신뢰도 부여):** "현재 `24,512`명이 테스트에 참여했어요" (숫자는 포인트 컬러로만 강조)
        
    - **선택 닉네임 입력:** CTA 바로 위에 선택 입력 필드로 배치합니다. 닉네임은 브라우저 세션에만 저장하며 서버에 저장하지 않습니다.

    - **CTA 버튼:** 화면 하단에 고정된 검정 버튼. (Label: **테스트 시작하기**)
        
    - **버튼 액션:** 클릭 시 `SessionStorage` 초기화 후 즉시 `/test` 라우트로 페이드 아웃 전환.
        

### 3.2. 테스트 진행 화면 (Test Progress Page)

오직 문항과 선택지에만 집중할 수 있도록 여백(Whitespace)을 극대화한 화면입니다.

- **[Top] 진행 상태바 (Progress Indicator)**
    
    - 뒤로 가기 버튼 (`<`)
        
    - **상태 표시:** `Phase 1 (탐색기)` 등 현재 단계 표시.
        
    - **Progress Bar:** 얇고 세련된 막대그래프로 진행률(%) 표시. 화면을 가로지르며 부드럽게 차오르는 애니메이션. (예: 5/20)
        
- **[Center] 딜레마 문항 카드**
    
    - **문항 텍스트:** `display.promptLines`를 기준으로 2줄 구조로 배치합니다. 진행률 아래 큰 빈 공간을 두지 않고, 질문 카드와 선택지 2개가 모바일 한 화면 안에 자연스럽게 들어오도록 합니다.
        
    - `display.promptHighlights`에 포함된 핵심어는 `<strong>`으로 강조하되, 현재 모노톤 UI에서는 핵심어 색상을 과하게 바꾸지 않습니다. 코랄 포인트는 진행률과 선택 상태 중심으로 사용합니다.

    - 문항이 길어질 경우 가독성을 위해 행간과 최대 폭을 안정적으로 유지합니다.
        
- **[Bottom] 양자택일 카드 버튼 영역**
    
    - 세로로 배치된 2개의 compact 카드형 버튼 (A 선택지 / B 선택지).

    - 각 카드는 `A/B 원형 뱃지 + 짧은 선택지 라벨 + 한 문장 설명` 구조로 렌더링합니다.

    - 선택지 라벨은 `display.choices.*.label`, 설명은 `display.choices.*.body`, 강조어는 `display.choices.*.highlights`를 사용합니다.
        
    - **버튼 디자인:** 흰색 카드, 얇은 회색 border, A/B 배지, 라벨, 한 문장 설명을 사용합니다. 선택/포커스 상태에서만 코랄 포인트를 사용합니다.
        
    - **인터랙션:** 버튼 클릭 시 해당 카드가 살짝 눌리는(Scale down) 애니메이션 후 다음 문항으로 전환합니다.
        

### 3.3. 로딩 및 분석 화면 (Loading & Interstitial)

20번째 마지막 문항을 선택한 직후, 서버(API)에서 결과를 연산하는 동안 노출되는 화면입니다. 텐션을 유발합니다.

현재 로컬 MVP에서는 `/analysis`를 별도 라우트로 만들지 않고 `/test` 화면 내부의 임시 로딩 상태로 구현합니다. 실제 MVP에서 공유/SEO/분석 이벤트를 더 세밀하게 다룰 필요가 생기면 `/analysis` 라우트로 분리합니다.

- **[Center] 분석 애니메이션**
    
    - "당신의 가치관 데이터를 분석 중입니다..." 텍스트 노출.
        
    - 선택했던 키워드들(공공 안전, 성과, 평등, 자유 등)이 스크린에 빠르게 떠올랐다가 사라지는 타이포그래피 애니메이션.
        
    - 진행률 표기: `0% -> 100%` (약 2~3초간 지속하여 신뢰감과 기대감 부여)
        
- **[Action]**
    
    - 100% 도달 시 `/result/[id]` 페이지로 전환.

### 3.3.1. 오류 및 복구 상태

API 실패나 세션 손상 시에도 사용자가 막다른 화면에 갇히지 않도록 아래 상태를 제공합니다.

| 상황 | 화면 메시지 방향 | 액션 |
|---|---|---|
| 다음 문항 로드 실패 | 다음 문항을 불러오지 못했다는 짧은 안내 | `다시 시도`, `처음부터 시작` |
| 응답 이력 검증 실패 | 진행 정보가 올바르지 않다는 안내 | `테스트 다시 시작` |
| 결과 계산 실패 | 결과를 계산하지 못했다는 안내 | `다시 시도`, `처음부터 시작` |
| 결과 데이터 없음 | 저장된 결과를 찾을 수 없다는 안내 | `테스트 다시 시작` |
| 결과 이미지 에셋 없음 | 캐릭터 없이 결과명, 결과 코드, 게이지 중심으로 표시 | 별도 액션 없음 |

오류 메시지는 길게 설명하지 않고 복구 행동을 우선합니다.
        

### 3.4. 최종 결과 화면 (Result Page - SNS 공유 최적화)

본 서비스의 핵심이자 바이럴의 시작점입니다. 현재 로컬 MVP는 공유 포스터보다 앱 안의 결과 대시보드처럼 읽히는 compact 카드 구조를 우선합니다.

- **[Summary Card] 결과 요약 카드**
    
    - _이 영역은 유저가 '이미지 저장'을 눌렀을 때 캡처 대상이 됩니다. 화면에서는 고정 9:16 비율을 강제하지 않습니다._
        
    - **결과 타이틀:** "나의 사회적·정치적 가치관은" / **[광장의 따뜻한 연대자]** (가장 눈에 띄게 강조)
        
    - **결과 코드:** `CTMO-S` 같은 코드를 작은 pill로 표시합니다. 타입 코드 네 글자를 대형 블록으로 반복하지 않습니다.
        
    - **한 줄 요약:** 결과 유형의 핵심 문장을 제목 바로 아래에 배치합니다.
        
    - **스탯 게이지 (4 Axis):**
        
        - 양극단이 표시된 4개의 가로 막대그래프 (예: `[C 공동체] 80% ■■■■□□ [I 개인] 20%`)
            
        - 게이지는 `resultViewModel.axisGauges`의 완화된 표시값을 사용합니다.
            
    - **워터마크:** 하단 조그맣게 `Poldigm.com` 텍스트를 표시합니다.
        
- **[Scroll Area 1] 상세 해석 영역**
    
    - **원칙 고수형/상황 조정형 상세 설명:** `desc_S` 또는 `desc_M` 텍스트 출력. 가독성을 위해 2~3문장 단위로 단락 구분.
        
- **[Scroll Area 2] 궁합 영역 (Chemistry)**
    
    - 좌우 반분된 카드 UI 레이아웃.
        
    - **잘 맞는 유형:** 유형 코드 또는 짧은 라벨
        
    - **다르게 보는 유형:** 유형 코드 또는 짧은 라벨
        
- **[Bottom] 공유 및 액션 영역**
    
    - 결과 상세/궁합 아래 3등분 버튼 그룹으로 배치합니다.
        
    - `[이미지 저장]` 버튼
        
    - `[링크 복사]` 버튼
        
    - `[다시 하기]` 버튼

- **[운영 제약]**

    - 결과 페이지는 검색 색인 대상이 아니므로 현재 로컬 MVP에서도 `noindex,follow` 메타를 적용합니다.

    - 광고를 적용하더라도 결과 요약과 공유 CTA 아래에만 배치하며, 캡처 영역 안에는 광고를 넣지 않습니다.

    - 테스트 진행 화면과 분석/로딩 화면에는 광고를 넣지 않습니다.
        

## 4. 모바일 UI 디자인 체크리스트 (디자이너 & 개발자 참고용)

1. **가독성:** 모바일 야외 환경(햇빛 아래)에서도 잘 보이도록 텍스트와 배경의 명도 대비를 충분히 확보할 것.
    
2. **터치 영역:** 테스트 문항 선택지 버튼의 터치 영역을 최소 `세로 120px` 이상 확보하여, 한 손(엄지손가락)으로 쾌적하게 탭할 수 있도록 설계할 것.
    
3. **색상 절제:** 기본 팔레트는 블랙, 그레이, 화이트를 사용하고 강조색은 코랄 한 가지로 제한합니다. 과한 그라데이션, 글로우, 네온 효과는 사용하지 않습니다.
    
4. **이미지 생성(html2canvas) 퀄리티:** 결과 화면을 이미지로 다운로드할 때 폰트가 깨지거나 CSS 레이아웃이 어긋나지 않는지 여러 모바일 디바이스에서 집중 QA 필수.

5. **접근성:** 선택지 카드는 키보드 포커스 상태가 보여야 하며, 텍스트 대비는 WCAG AA 수준을 목표로 합니다.

6. **긴 텍스트:** 문항이나 선택지가 길어져도 버튼 높이 변화로 레이아웃이 깨지지 않도록 충분한 줄간격과 최대 폭을 유지합니다.

7. **상태 표시:** 로딩, disabled, 재시도, 결과 없음 상태는 모두 실제 화면으로 구현합니다.

## 5. 현재 구현 반영

현재 로컬 MVP 화면 구현은 아래 컴포넌트 구조를 사용합니다.

| 화면/요소 | 구현 파일 |
|---|---|
| 공통 헤더 | `src/lib/components/layout/AppHeader.svelte` |
| 랜딩 | `src/routes/+page.svelte` |
| 테스트 진행 | `src/routes/test/+page.svelte` |
| 선택지 카드 | `src/lib/components/test/ChoiceCard.svelte` |
| 진행률 | `src/lib/components/test/ProgressBar.svelte` |
| 결과 페이지 | `src/routes/result/[id]/+page.svelte` |
| 결과 캡처 영역 | `src/lib/components/result/ResultCapture.svelte` |
| 축 게이지 | `src/lib/components/result/AxisGauge.svelte` |
| 캐릭터 fallback 컴포넌트 | `src/lib/components/result/CharacterImage.svelte` (현재 기본 결과 카드에서는 미표시) |
| 하단 액션 | `src/lib/components/result/BottomActions.svelte` |

`AxisGauge.svelte`는 `resultViewModel.axisGauges`의 표시용 값을 그대로 사용해 왼쪽/오른쪽 퍼센트를 함께 렌더링합니다. 원점수 `scores`를 다시 퍼센트로 계산하지 않으며, 극단값도 `88 : 12`처럼 완화된 게이지로 보여줍니다.

검증된 UI 상태:

- 랜딩 렌더링
- 20문항 완료 후 결과 이동
- 진행 중 새로고침 복구
- 결과 페이지 새로고침 복구
- sessionStorage fallback
- 결과 없음 CTA
- desktop/mobile 및 시스템 Chrome E2E
