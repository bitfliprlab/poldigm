# 📄 01_PRD: 제품 요구사항 정의서 (Product Requirements Document)

## 1. 프로젝트 개요 (Overview)

- **프로젝트명:** 폴다임 (Poldigm)
    
- **프로덕트 형태:** 모바일 웹 기반 성향 진단 플랫폼 (Mobile-first Web App)
    
- **핵심 요약:** 기존의 지루하고 평면적인 성향 테스트를 탈피하여, 적응형 분기 알고리즘(CAT)을 통해 개인의 사회적·정치적 가치관을 16가지 기본 유형과 강도 태그(S/M)로 입체적으로 분석해 주는 바이럴 테스트.
    

## 2. 대상 사용자 및 문제 정의 (Target & Problem)

### 2.1. 타겟 유저

- **Primary:** 정치·사회 문제에 관심은 있으나 극단적인 양당 체제나 기존의 낡은 이념 프레임에 소속감을 느끼지 못하는 2030 세대.
    
- **Secondary:** MBTI 등 심리테스트 결과를 SNS(인스타그램, X)에 공유하며 자아를 표현하기 즐기는 일반 대중.
    

### 2.2. 해결하고자 하는 문제 (Pain Points)

- 기존 테스트의 지루함: "매우 동의함 ~ 매우 비동의함" 식의 5점 척도 설문은 몰입도가 떨어짐.
    
- 애매한 결과: 혼합 응답이나 상황 조정형 사용자를 단순한 '회색지대'로 취급하여 결과에 대한 공감대가 낮음.
    
- 바이럴 요소 부재: 결과를 타인과 공유하거나 논의하고 싶게 만드는 네이밍 및 시각적 매력이 부족함.
    

## 3. 제품의 핵심 가치 (Core Values & USP)

1. **몰입형 딜레마 (Zero 'Neutral'):** '보통' 선택지를 없애고 가혹한 양자택일 딜레마를 강제하여 유저의 진짜 가치관을 도출.
    
2. **적응형 분기 알고리즘 (CAT):** 유저의 이전 대답에 따라 다음 문항의 뎁스(Depth)가 달라지는 꼬리 문항 방식을 통해 게임 같은 경험 제공.
    
3. **입체적 결과 세분화:** 4가지 기준 축(C/I, T/P, M/E, O/L) 조합에 더해, 원칙 고수형(-S)과 상황 조정형(-M) 태그를 부여하여 혼합 응답 사용자의 차이를 더 정확히 설명.
    

## 4. 주요 기능 요구사항 (Functional Requirements)

### 4.1. 온보딩 및 진입 (Zero Friction)

- **무인증 시스템:** 회원가입이나 로그인 절차 없이 랜딩 페이지에서 [테스트 시작] 버튼 클릭 즉시 1번 문항으로 진입해야 함.
    
- **소셜 프루프 노출:** 랜딩 페이지에 '현재 참여자 수'를 동적으로 노출하여 신뢰도와 호기심 유발.
    
- **선택 닉네임:** 사용자는 결과 화면 개인화를 위해 닉네임을 선택 입력할 수 있으나, 미입력 시에도 테스트와 결과 확인이 가능해야 함.
    

### 4.2. 테스트 진행 (Test Flow)

- **상태 표시:** 상단 프로그래스 바를 통해 진행률을 직관적으로 제공.
    
- **문항 노출 로직 (Phase 1 → Phase 2):**
    
    - Phase 1 (탐색기): 8문항. 기본 성향 파악.
        
    - Phase 2 (심층기): 12문항. Phase 1 결과에 기반한 한계 상황 제시. MVP의 전체 테스트 길이는 Phase 1 8문항 + Phase 2 12문항, 총 20문항으로 고정.
        
- **세션 유지:** 새로고침 시에도 이전 응답 기록이 날아가지 않도록 `sessionStorage` 기반 상태 관리. 브라우저 탭을 닫으면 테스트 상태는 소멸하는 것을 기본 정책으로 함.
    

### 4.3. 결과 도출 및 공유 (Result & Viral)

- **결과 연산:** 서버 사이드에서 유저의 응답 데이터를 기반으로 가중치를 합산하여 최종 유형(예: `CTMO-S`) 도출.
    
- **결과지 구성:**
    
    - 유형별 직관적 타이틀 (예: 공동체의 원칙 설계자), 결과 코드 pill, 한 줄 요약.
        
    - 4개 축에 대한 표시용 경향 퍼센트(%) 게이지. 원점수 `scores`를 그대로 노출하지 않고 `resultViewModel.axisGauges`의 완화된 표시값을 사용한다.
        
    - S(원칙 고수형)/M(상황 조정형) 태그에 따른 가치관 설명 텍스트.
        
    - 잘 맞는 유형/다르게 보는 유형 매칭.
        
- **공유 기능:**
    
    - 결과 요약 카드 이미지 다운로드 기능 (html2canvas 등 활용). 실제 MVP에서 공유 성과를 확인한 뒤 9:16/16:9 전용 이미지 확장을 검토.
        
    - 클립보드 링크 복사 버튼.
        

## 5. 비기능 요구사항 및 기술 스택 (Non-functional)

|**구분**|**요구사항 및 사용 기술**|
|---|---|
|**프론트엔드**|SvelteKit, CSS 변수 + 컴포넌트 scoped CSS (모바일 최적화 반응형 UI)|
|**백엔드/인프라**|Cloudflare Pages + Pages Functions/Workers (정적 호스팅 무료 기반, API는 사용량 기준 과금)|
|**데이터베이스**|Supabase (`test_results` 결과 통계 저장 전용, 클라이언트 직접 접근 금지)|
|**보안 및 알고리즘**|문항 풀(`questions.ko-KR.ts`) 및 점수 연산 로직은 SvelteKit 서버 전용 영역에 은닉 (클라이언트 노출 차단)|
|**어뷰징/통계**|Cloudflare Turnstile 도입, IP 원문 미저장, 국가 코드는 통계용으로만 저장|
|**언어 정책**|MVP는 `ko-KR` 단일 지원. 다국어는 후속 확장으로 검토하되 DB와 파일 구조에 locale 확장 여지만 확보|
|**SEO/공유/광고**|실제 MVP 배포 전 title/description, OG/Twitter Card, robots/sitemap, 결과 페이지 `noindex`, AdSense 적용 가능성을 점검. 로컬 MVP에서는 광고 스크립트를 넣지 않음|

## 6. 마일스톤 및 일정 (Schedule)

총 8주 스프린트 기준으로 진행하되, 구현 단계는 **로컬 MVP → 공개 MVP → 운영 MVP**로 분리합니다.

### 6.1. 단계 정의

| 단계 | 목표 | 포함 | 제외 |
|---|---|---|---|
| 로컬 MVP | 외부 인프라 없이 테스트 전체 플로우 검증 | SvelteKit 로컬 구동, 20문항 진행, 서버 알고리즘, mock 저장, 결과 화면 | Supabase 실제 연결, Cloudflare 배포, 실제 Turnstile, 광고, 운영용 OG |
| 공개 MVP | 실제 도메인에서 소규모 사용자 검증 | Cloudflare 배포, Supabase 저장, Turnstile, 기본 SEO/OG, 약관/개인정보 페이지 | 본격 광고 최적화, 관리자 페이지, 다국어 |
| 운영 MVP | 트래픽/공유/수익화 실험 | SEO 검증, 결과 공유 최적화, 제한적 광고 슬롯, 통계 리포트 기반 개선 | 대규모 커뮤니티/앱 확장 |

### 6.2. 8주 실행안

- **Week 1-2 (기획/문서화):** PRD 확정, MVP 필수 44개 알고리즘 슬롯과 132개 후보 문항 풀 방향성 설계, 20문항 노출용 가중치 로직 설계 완료, 16개 유형 텍스트 작성.
    
- **Week 3-4 (디자인/법무):** UI/UX 와이어프레임 설계, 디자인 시스템 및 캐릭터 일러스트 제작, 상표권 출원.
    
- **Week 5 (로컬 MVP 개발):** SvelteKit 프론트엔드 퍼블리싱, 분기 로직 API 연동, mock 저장/검증, 로컬 결과 화면 구현.

- **Week 6 (공개 MVP 전환):** Supabase DB 스키마 및 RLS 설정, Cloudflare Pages preview/production 배포 확인, Turnstile 검증, 실제 도메인 연결.
    
- **Week 7 (QA/테스트):** 사내 알파 테스트, 모바일 크로스 브라우징 테스트, 알고리즘 득점 밸런스 조정.
    
- **Week 8 (런칭/마케팅):** 상용 서버 배포(Cloudflare), 초기 시드 타겟 커뮤니티 배포, SNS 공유 트래픽 모니터링, SEO/OG 미리보기 점검. 광고는 기본 비활성화하고, AdSense는 승인 가능성 검토 후 결과 페이지 하단 1개 슬롯부터 별도 실험.

## 7. 현재 구현 반영 상태

2026-06-22 기준 로컬 MVP는 구현 완료 상태입니다.

| 항목 | 구현 상태 |
|---|---|
| 프레임워크 | SvelteKit + TypeScript |
| 로컬 실행 | `npm run dev`, `http://127.0.0.1:5173/` |
| 페이지 | 랜딩 `/`, 테스트 `/test`, 결과 `/result/[id]` |
| API | `/api/next-question`, `/api/submit-result` |
| 문항 | 서버 전용 44개 알고리즘 슬롯, 132개 후보 문항, 사용자 노출 20문항 |
| 결과 | 16개 유형과 `S/M` 강도 태그 산출 |
| 저장 | Supabase 없이 mock 메모리 저장소 |
| 검증 | `npm run verify`, `npm run verify:ci`, `npm run test:e2e`, `npm run test:e2e:chrome` 통과 |

공개 MVP 전환 전 남은 범위는 Supabase 저장 adapter, Cloudflare Pages 실제 배포, 실제 Turnstile, 실제 도메인/OG/SEO, 운영용 법무 정보 확정입니다.

현재 로컬 MVP 결과 화면은 캐릭터 일러스트 없이 결과명, 결과 코드, 한 줄 설명, 4축 게이지, 상세 해석, 궁합, 이미지 저장/링크 복사/다시 하기 액션으로 구성합니다. 캐릭터 일러스트와 결과별 OG 이미지는 실제 MVP 이후 공유 성과를 보고 확장합니다.
