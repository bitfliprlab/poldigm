# 📄 30_시스템_아키텍처_설계서

## 1. 문서 개요

본 문서는 폴다임(Poldigm) 서비스의 초기 운영 비용 절감, 바이럴 트래픽 대응, 보안(알고리즘 유출 및 어뷰징 방지)을 달성하기 위한 **전체 시스템 아키텍처 및 데이터 흐름도**를 정의합니다. 개발팀은 본 설계서를 바탕으로 인프라 환경을 구축하고 프론트엔드/백엔드 통신 규약을 확립합니다.

## 2. 전체 시스템 구성도 (High-Level Architecture)

Poldigm MVP는 별도의 상시 백엔드 서버(Lightsail/EC2 등) 없이, **Cloudflare Pages와 Pages Functions/Workers 런타임, 서버리스 DB를 결합한 엣지 중심 아키텍처**를 사용합니다.

이 선택은 현재 기능이 짧은 API 처리(문항 분기, 점수 재계산, Turnstile 검증, Supabase 적재)에 집중되어 있기 때문입니다. 장기 실행 작업, 큐/배치, 관리자 백엔드, WebSocket 같은 요구가 생기는 단계에서는 Lightsail/EC2 또는 별도 워커 인프라를 추가 검토합니다.

> 로컬 MVP 구현 예외: Supabase, Cloudflare Pages, Turnstile, 실제 도메인 없이 `http://127.0.0.1:5173`에서 구동하는 MVP는 `33_로컬_MVP_구현_계획서`를 우선합니다. 본 문서는 실제 MVP 전환 및 배포 단계의 기준 아키텍처입니다.

```text
[ Client / Browser ]  <--- (1) UI 렌더링 & 상태 관리 --->  [ Cloudflare Edge Network ]
  - SvelteKit (Frontend)                                     - SvelteKit (Server-side API)
  - HTML/CSS/JS (CSS 변수 기반 스타일)                         - /api/next-question
  - SessionStorage (상태 캐싱)                                - /api/submit-result
  - Turnstile (봇 검증 위젯)                                  - 서버 문항 모듈 은닉
          |                                                             |
          | (2) 봇 검증 토큰 발급                                          | (3) 최종 결과 Insert
          v                                                             v
[ Cloudflare Turnstile API ]                               [ Supabase (PostgreSQL) ]
  - 토큰 유효성 검증                                           - test_results 테이블
                                                               - RLS (Row Level Security) 적용
```

## 3. 핵심 컴포넌트 및 역할 명세

### 3.1. 프론트엔드 (Client-side)

- **기술 스택:** SvelteKit, CSS 변수 + 컴포넌트 scoped CSS
    
- **역할:**
    
    - 사용자 UI 렌더링 및 페이지 전환 (SPA 라우팅).
        
    - `sessionStorage`를 활용한 테스트 진행 상태(현재 문항, 누적 선택지) 임시 저장 (이탈 후 재접속 시 복구용).
        
    - html2canvas 라이브러리를 활용한 결과 화면 이미지 렌더링 및 다운로드.
        
    - Cloudflare Turnstile 위젯을 백그라운드에서 실행하여 사람임을 증명하는 토큰 발급.
        

### 3.2. 엣지 서버 API (Server-side)

- **기술 스택:** Cloudflare Pages + Pages Functions/Workers 런타임, SvelteKit `adapter-cloudflare`, SvelteKit `+server.ts` API routes
    
- **역할:**
    
    - **알고리즘 은닉:** 44개 알고리즘 슬롯과 132개 후보 문항 풀(`questions.ko-KR.ts`)의 점수 배점/분기 조건 로직을 클라이언트 브라우저에서 절대 볼 수 없도록 서버 메모리에서만 처리.
        
    - **적응형 문항 서빙 (`/api/next-question`):** 클라이언트가 보낸 응답 기록을 바탕으로 다음 문항을 계산하고, `prompt`, `choices`, `display`, `progress` 같은 공개 표시 데이터만 프론트엔드로 전달합니다. `scoreEffect`, `branchCondition`, `intensityEffect` 같은 내부 배점/분기 필드는 응답에서 제외합니다.
        
    - **최종 결과 연산 (`/api/submit-result`):** 조작 방지를 위해 클라이언트가 보낸 전체 응답 기록을 서버에서 다시 한번 합산하여 최종 성향 코드(예: `CTMO-S`) 도출.
        
    - Turnstile 토큰 검증 후 서버 환경변수에 보관된 Supabase 권한으로 데이터 전송.
        

### 3.3. 데이터베이스 (Database)

- **기술 스택:** Supabase (PostgreSQL 기반 BaaS)
    
- **역할:**
    
    - 최종 산출된 테스트 결과 통계 및 익명 유저 데이터 수집.
        
    - 클라이언트는 Supabase와 직접 통신하지 않고, SvelteKit 서버 API가 검증 후 Supabase에 적재합니다.
        

## 4. 핵심 데이터 흐름도 (Data Flow)

### Flow 1: 적응형 테스트 진행 (Phase 1 → Phase 2)

1. **[Client]** 사용자가 1번 문항에서 'A'를 선택합니다.
    
2. **[Client]** 로컬 상태(`sessionStorage`)에 응답 내역을 추가하고, `/api/next-question`으로 지금까지의 누적 응답 배열(`history`)을 `POST` 요청합니다.
    
3. **[Edge Server]** 누적 응답을 분석하여 Phase 1이 끝났는지 판단합니다.
    
4. **[Edge Server]** Phase 2 진입 시, 각 축의 Phase 1 결과에 맞는 분기 조건(예: `C`, `I`, `BALANCED`)과 `branchOrder`에 해당하는 문항을 서버 전용 문항 모듈에서 찾습니다.
    
5. **[Edge Server]** 해당 문항의 공개 표시 데이터만 [Client]로 응답(Response)합니다. 문항 본문, 선택지, 2줄 표시 모델, 강조 키워드, 진행률은 포함하고 점수/분기 정보는 제외합니다.
    
6. **[Client]** 전달받은 새 문항을 화면에 부드럽게 렌더링합니다.
    

### Flow 2: 테스트 완료 및 결과 도출

1. **[Client]** 20번째 마지막 문항을 선택합니다.
    
2. **[Client]** 백그라운드에서 Turnstile 토큰을 생성합니다.
    
3. **[Client]** 전체 20개의 응답 배열(`history`)과 Turnstile 토큰을 `/api/submit-result`로 전송합니다.
    
4. **[Edge Server]** Turnstile API를 호출하여 봇(매크로) 여부를 1차 검증합니다.
    
5. **[Edge Server]** 봇이 아니라면, 서버의 배점표를 기준으로 최종 점수를 연산하고 `CTMO-S` 등의 최종 결과를 확정합니다.
    
6. **[Edge Server]** 결과 데이터를 Supabase에 `INSERT` 요청합니다.
    
7. **[Edge Server]** DB 저장 완료 후, [Client]에게 최종 결과 코드 및 맵핑 텍스트를 반환합니다.
    
8. **[Client]** 반환받은 데이터를 바탕으로 결과 페이지를 렌더링합니다.
    

## 5. 인프라 및 보안 특장점 (Security & Scaling)

본 아키텍처는 초기 비용을 낮게 유지하면서, 트래픽 증가에 따라 Cloudflare Workers 및 Supabase 플랜을 단계적으로 확장할 수 있도록 설계되었습니다.

1. **Usage-based Edge Scaling:**
    
    - HTML/CSS/JS 정적 파일은 Cloudflare Pages의 CDN 엣지 네트워크에서 제공됩니다.
        
    - `/api/*` 요청은 Pages Functions/Workers 사용량 기준으로 계산됩니다. 따라서 한 사용자가 20문항을 모두 진행할 경우 약 20~21회의 API 호출이 발생한다는 점을 트래픽 산정에 반영합니다.
        
    - MVP는 무료/저비용 구간에서 시작하되, 공개 런칭 또는 바이럴 캠페인 전에는 Cloudflare Workers 유료 플랜과 Supabase Pro 전환 여부를 점검합니다.
        
2. **Lightsail/EC2 대비 운영 단순성:**
    
    - Poldigm MVP는 상시 실행 서버보다 짧은 요청-응답 API가 중심이므로, OS 패치, Nginx, 프로세스 매니저, 인스턴스 스케일링을 직접 운영하지 않는 Cloudflare Pages Functions 구성이 적합합니다.
        
    - 장기 실행 작업, 큐/배치 처리, 별도 관리자 서버가 필요해지는 단계에서는 Lightsail/EC2 또는 전용 워커 인프라를 별도 검토합니다.
        
3. **무인증 어뷰징 차단 (Server-validated Insert & Turnstile):**
    
    - 비로그인 서비스의 취약점인 '무한 결과 제출 테러(DB 스팸)'를 막기 위해 **Cloudflare Turnstile**을 적용합니다. 유저에게 거슬리는 신호등/횡단보도 퀴즈(reCAPTCHA)를 풀게 하지 않고, 브라우저 환경을 은밀히 분석하여 봇을 차단합니다.
        
    - 클라이언트는 Supabase에 직접 `INSERT`하지 않습니다. SvelteKit 서버 API가 Turnstile, 플레이 시간, 응답 기록 정합성을 검증한 뒤 Supabase에 적재합니다.
        
    - **Supabase RLS 설정:** RLS는 방어 계층으로 유지하며, 클라이언트 키가 노출되더라도 기존 데이터를 읽거나(`SELECT`), 수정/삭제(`UPDATE`, `DELETE`)할 수 없도록 제한합니다.
        
4. **지적재산권(알고리즘) 보호:**
    
    - 모든 심화 문항 데이터와 점수 연산 로직이 SvelteKit 백엔드(`+server.ts`)에 위치하므로, 크롬 개발자 도구(F12)를 통해서도 서비스의 핵심 노하우를 빼낼 수 없습니다.

## 6. 로컬 MVP → 실제 MVP 전환 체크리스트

`33_로컬_MVP_구현_계획서` 기준으로 로컬 MVP가 완료된 뒤, 실제 MVP 배포 전 아래 항목을 전환합니다.

| 구분 | 로컬 MVP | 실제 MVP 전환 기준 |
|---|---|---|
| SvelteKit adapter | `adapter-cloudflare` 적용 | Cloudflare Pages preview/production 빌드 확인 |
| 결과 저장 | `mock-results.ts` 메모리 저장 | Supabase insert adapter로 교체 |
| 결과 조회 | `+page.server.ts`에서 mock 조회 | Supabase `test_results` 조회 또는 공개 공유 정책에 맞는 조회 adapter 적용 |
| 봇 검증 | `mock-turnstile.ts` | Cloudflare Turnstile server verification 적용 |
| 환경변수 | 빈 Supabase/Turnstile secret 허용 | `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `TURNSTILE_SECRET_KEY`, `PUBLIC_TURNSTILE_SITE_KEY` 설정 |
| Abuse 기준 | `playTimeSec < 20` | 동일 기준 유지. 운영 중 로그 기반으로 조정 시 API/DB/정책 문서 동시 갱신 |
| SEO/OG | 로컬에서는 최소 구현 | `34_SEO_및_광고_노출_정책서` 기준 메타, OG, robots, sitemap 적용 |
| 법무 페이지 | `/terms`, `/privacy` 최소 정적 페이지 제공 | 운영 주체, 문의처, 실제 수집 항목 반영 |

전환 후에도 문항 데이터 스키마, 점수 계산 알고리즘, API request/response 계약, 결과 매핑 구조는 바꾸지 않는 것을 원칙으로 합니다.

## 7. 현재 구현 반영

현재 로컬 MVP 아키텍처는 외부 인프라 없이 아래 구조로 동작합니다.

| 계층 | 현재 구현 |
|---|---|
| 앱 서버 | SvelteKit dev server |
| 배포 adapter | `@sveltejs/adapter-cloudflare` |
| API | SvelteKit `+server.ts` |
| 문항/결과 데이터 | 서버 전용 TS 모듈 |
| 결과 저장 | `mock-results.ts` 메모리 Map |
| 봇 검증 | SvelteKit dev 런타임에서만 `mock-turnstile.ts`의 `local-mock-token` 허용 |
| 결과 조회 | `/result/[id]/+page.server.ts`에서 mock 조회 후 client fallback |
| 공유/저장 | 클립보드 복사, `html2canvas` 브라우저 동적 import |

현재 코드에는 Cloudflare adapter가 적용되어 있습니다. 아직 Cloudflare Pages 실제 배포, Supabase, 실제 Turnstile, 실제 도메인은 연결하지 않았습니다. 이 문서의 실제 MVP 아키텍처는 다음 단계 전환 기준입니다.
