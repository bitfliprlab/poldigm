# 📄 10_문항_및_분기_로직_설계서

## 1. 문서 개요

본 문서는 폴다임(Poldigm) 서비스의 핵심인 '적응형 분기 알고리즘(CAT)'을 구현하기 위한 문항 풀(Pool)의 구조, 단계별 분기 조건, 그리고 최종 결과 도출을 위한 가중치 배점표를 정의합니다. 백엔드 및 프론트엔드 개발자는 본 설계서의 로직을 기준으로 API 및 데이터 스키마를 구성해야 합니다.

### 1.1. 현재 구현 기준

2026-06-22 기준 로컬 MVP는 본 문서의 규칙을 아래 파일에 구현했습니다.

| 구분 | 구현 파일 |
|---|---|
| 서버 전용 문항 풀 | `src/lib/server/data/questions.ko-KR.ts` |
| 다음 문항 계산 | `src/lib/server/algorithm/questions.ts` |
| 점수/S-M 계산 | `src/lib/server/algorithm/scoring.ts` |
| 이력/문항 풀 검증 | `src/lib/server/algorithm/validation.ts` |
| 테스트 fixture | `tests/helpers/fixtures.ts` |
| 알고리즘 테스트 | `tests/algorithm/questions.test.ts`, `tests/algorithm/scoring.test.ts`, `tests/algorithm/type-coverage.test.ts` |
| 문항 품질 테스트 | `tests/algorithm/question-quality.test.ts` |

현재 구현은 44개 알고리즘 슬롯, 132개 후보 문항, 20문항 고정 슬롯 노출, seed 기반 슬롯 내 variant 선택, 16개 기본 유형, `S/M` 강도 태그, `BALANCED` 분기, invalid history 차단을 자동 테스트로 검증합니다.

## 2. 알고리즘 코어 로직 및 점수 체계

폴다임의 성향 측정은 총 4개의 축(C/I, T/P, M/E, O/L)에 대해 각각 독립적으로 연산됩니다.

각 축의 최대 획득 가능 점수는 **100점**입니다. 최종 알파벳은 축별 점수로 결정하고, 강도 태그(S/M)는 Phase 2에서 해당 성향의 원칙을 얼마나 고수했는지를 기준으로 결정합니다.

### 2.1. 문항 구성 및 배점 (축당 5문항, 총 20문항 노출 기준)

- **Phase 1 (탐색기):** 축당 2문항 배정. (총 8문항)
    
    - 목적: 유저의 기본 성향(방향키) 판단.
        
    - 배점: 문항당 20점. (Phase 1 완료 시 최대 40점 획득)
        
- **Phase 2 (심층기):** 축당 3문항 배정. (총 12문항)
    
    - 목적: 한계 딜레마를 통한 신념의 강도(S/M) 판별.
        
    - 배점: 문항당 20점. (Phase 2 완료 시 최대 60점 획득)
        

### 2.2. 분기(Branching) 발동 조건

Phase 1(탐색기)의 2문항 응답 결과에 따라 Phase 2(심층기)의 문항 풀이 달라집니다.

- **조건 A (한쪽 쏠림):** Phase 1에서 특정 지표를 2번 모두 선택 (예: C 40점 / I 0점)
    
    - 👉 **분기 로직:** 해당 지표의 한계를 시험하는 '한계 검증 문항 세트 A' 노출.
        
- **조건 B (혼합 선택):** Phase 1에서 양쪽 지표를 1번씩 선택 (예: C 20점 / I 20점)
    
    - 👉 **분기 로직:** 양극단의 가치가 정면충돌하는 '혼합 응답용 밸런스 문항 세트 B' 노출.

### 2.3. MVP 문항 노출 순서와 variant 선택

MVP에서는 결과 재현성과 QA 편의성을 위해 알고리즘 슬롯 순서를 랜덤으로 섞지 않습니다. 서버는 아래 고정 순서에 따라 다음 슬롯을 결정한 뒤, 해당 슬롯 안의 후보 문항 중 하나를 `questionSeed` 기반으로 선택합니다.

| 노출 순번 | 단계 | 축 | 선택 규칙 |
|---|---|---|---|
| 1 | Phase 1 | C/I | `phase=1`, `axis=C_I`, `commonOrder=1` |
| 2 | Phase 1 | T/P | `phase=1`, `axis=T_P`, `commonOrder=1` |
| 3 | Phase 1 | M/E | `phase=1`, `axis=M_E`, `commonOrder=1` |
| 4 | Phase 1 | O/L | `phase=1`, `axis=O_L`, `commonOrder=1` |
| 5 | Phase 1 | C/I | `phase=1`, `axis=C_I`, `commonOrder=2` |
| 6 | Phase 1 | T/P | `phase=1`, `axis=T_P`, `commonOrder=2` |
| 7 | Phase 1 | M/E | `phase=1`, `axis=M_E`, `commonOrder=2` |
| 8 | Phase 1 | O/L | `phase=1`, `axis=O_L`, `commonOrder=2` |
| 9~12 | Phase 2 | C/I, T/P, M/E, O/L | 각 축의 선택된 `branchCondition`, `branchOrder=1` |
| 13~16 | Phase 2 | C/I, T/P, M/E, O/L | 각 축의 선택된 `branchCondition`, `branchOrder=2` |
| 17~20 | Phase 2 | C/I, T/P, M/E, O/L | 각 축의 선택된 `branchCondition`, `branchOrder=3` |

Phase 2의 축 순서는 항상 `C_I → T_P → M_E → O_L`입니다. 예를 들어 C/I 축 Phase 1 결과가 C 40점이면 C/I 축의 Phase 2 문항은 `branchCondition='C'`이고, C 20점/I 20점이면 `branchCondition='BALANCED'`입니다.

문항 variant 원칙:

- 같은 슬롯의 후보 문항은 `axis`, `phase`, `branchCondition`, `commonOrder` 또는 `branchOrder`, `scoreEffect`, `intensityEffect`가 같아야 합니다.
- 사용자에게 보이는 `prompt`, `choices`, `display`만 다르게 작성합니다.
- 같은 `questionSeed`와 같은 history에서는 같은 후보 문항이 다시 선택되어야 합니다.
- 후보 선택은 세션 내 `copyFamily` 사용량을 먼저 보정하고, 최근 3문항의 `scenarioTag`, `toneTag` 반복을 피한 뒤, 전체 `scenarioTag`, `toneTag` 사용량과 `questionSeed`를 적용합니다.
- 서버 검증은 응답된 문항이 해당 시점에 도달 가능한 슬롯의 후보인지 확인합니다.

### 2.4. 서버 전용 문항 풀 최소 구성

`132개 후보 문항 풀`은 현재 MVP 구현 기준이며, MVP에서 실제 구현에 반드시 필요한 최소 알고리즘 슬롯 수는 아래와 같습니다. 각 축은 같은 구조를 가져야 하며, 서버는 각 세트에서 `branchOrder` 1~3을 순서대로 노출합니다.

| 축 | Phase 1 공통 문항 | Phase 2 왼쪽 지표 분기 | Phase 2 오른쪽 지표 분기 | Phase 2 BALANCED 분기 | 축별 최소 문항 |
|---|---:|---:|---:|---:|---:|
| C/I | 2 | 3 (`C`) | 3 (`I`) | 3 | 11 |
| T/P | 2 | 3 (`T`) | 3 (`P`) | 3 | 11 |
| M/E | 2 | 3 (`M`) | 3 (`E`) | 3 | 11 |
| O/L | 2 | 3 (`O`) | 3 (`L`) | 3 | 11 |
| **합계** | **8** | **12** | **12** | **12** | **44** |

즉, MVP 서버 전용 문항 파일은 최소 44개 알고리즘 슬롯을 포함해야 합니다. 유저 한 명에게 노출되는 문항은 이 중 20개 슬롯에서 선택된 20문항입니다. 현재 구현은 각 슬롯에 기본 문항 1개, V2 후보 1개, V3 후보 1개를 두어 총 132개 후보 문항을 제공합니다.

향후 후보 문항 풀을 더 확장할 때도 MVP와 같은 결정 규칙을 유지하려면, 같은 `axis + phase + branchCondition + branchOrder` 슬롯 안에서만 후보 문항을 추가하거나 A/B 테스트해야 합니다. 현재 운영 권장 기준은 슬롯당 3개 후보, 총 132개 후보 문항입니다.

### 2.5. BALANCED 분기 점수 및 강도 처리

`BALANCED` 분기는 Phase 1에서 양쪽 지표를 1회씩 선택한 축에만 사용합니다. 이 분기의 목적은 최종 알파벳 방향을 결정하는 것이며, 원칙 고수형 여부를 과도하게 부풀리지 않는 것입니다.

- `BALANCED` 문항도 A/B 선택에 따라 한쪽 지표에 `+20`을 부여합니다. 예: A는 C +20, B는 I +20.
- 단, `BALANCED` 문항의 선택은 S/M 판별용 원칙 고수 횟수에 포함하지 않습니다.
- 따라서 Phase 1에서 이미 혼합 선택을 보인 축은 최종 알파벳에는 영향을 주되, 해당 축 하나만으로 원칙 고수형 태그를 강화하지 않습니다.
- `BALANCED` 문항은 양극단의 장단점이 동시에 드러나도록 작성하며, 어느 선택지도 명백히 선하거나 악하게 보이지 않아야 합니다.

### 2.6. 문항 ID 명명 규칙

서버 검증과 QA 자동화를 위해 문항 ID는 아래 규칙을 따릅니다. ID는 로케일과 무관하게 안정적으로 유지하며, 문구를 수정해도 ID를 바꾸지 않습니다.

| 문항 유형 | ID 패턴 | 예시 |
|---|---|---|
| Phase 1 공통 문항 | `Q_1_{AXIS}_{commonOrder}` | `Q_1_C_I_1`, `Q_1_T_P_2` |
| Phase 2 한쪽 쏠림 분기 | `Q_2_{TARGET}_{branchOrder}` | `Q_2_C_1`, `Q_2_P_3` |
| Phase 2 밸런스 분기 | `Q_2_{AXIS_COMPACT}_BAL_{branchOrder}` | `Q_2_CI_BAL_1`, `Q_2_OL_BAL_3` |
| Variant 문항 | `{BASE_ID}_V{n}` | `Q_1_C_I_1_V2`, `Q_2_C_1_V2` |

- `AXIS`는 `C_I`, `T_P`, `M_E`, `O_L` 중 하나입니다.
- `AXIS_COMPACT`는 축 이름에서 `_`를 제거한 값입니다. 예: `C_I` → `CI`.
- `TARGET`은 `C`, `I`, `T`, `P`, `M`, `E`, `O`, `L` 중 하나입니다.
- `commonOrder`는 1 또는 2, `branchOrder`는 1, 2, 3만 허용합니다.
- variant suffix는 같은 슬롯의 후보 문항을 구분하기 위한 값이며, 알고리즘 슬롯 자체를 새로 만들지 않습니다.

### 2.7. 문항 작성 품질 기준

문항은 정치 진영 맞히기 퀴즈가 아니라 가치 충돌을 측정하는 도구입니다. 아래 기준을 지키지 못한 문항은 서버 데이터에 넣기 전에 다시 작성합니다. 현재 구현은 추상적인 정책 문장보다 사용자가 바로 떠올릴 수 있는 **생활 장면형 딜레마**를 우선합니다.

#### 전체 톤앤매너

Poldigm 문항은 **차갑고 직접적인 정치 성향 테스트**가 아니라, **생활 속 가치 판단을 통해 사회적·정치적 가치관이 드러나는 딜레마 테스트**처럼 읽혀야 합니다.

- 사용자가 특정 진영에 배정되는 느낌보다, “내가 어떤 의사결정 기준을 더 우선하는지”를 고르게 만듭니다.
- 문장은 신뢰감 있는 진단 도구처럼 차분해야 하지만, 행정문서나 정책 보고서처럼 딱딱하게 쓰지 않습니다.
- 아파트 단톡방, 공용 공간, 학교 행사, 동아리, 직장 보상, 공원 이용처럼 사용자가 이미 아는 장면에서 가치 충돌을 꺼냅니다.
- 선택지 라벨은 축 이름을 설명하는 문구가 아니라 `먼저 막기`, `동의 먼저`, `천천히 바꾸기`, `먼저 열어두기`, `기여대로`, `버틸 만큼`, `미리 정하기`, `해보고 고치기`처럼 사용자가 누를 수 있는 행동 기준으로 씁니다.
- Phase 2 한쪽 쏠림 분기에서는 `그대로 간다`와 `한발 물린다` 라벨을 사용해 원칙 유지와 제한적 양보의 차이를 짧게 보여줍니다.
- `상황입니다`, `기준입니다`, `찬성한다`, `반대한다` 같은 표현은 필요할 때만 사용하고 반복을 줄입니다.
- 실제 정당, 선거, 후보, 정치인, 특정 정책명, 진영명은 직접 언급하지 않습니다.
- A/B 중 한쪽이 더 착하거나 합리적으로 보이면 안 됩니다. 두 선택지 모두 나름의 명분과 비용이 있어야 합니다.
- “옳다/틀리다”, “우월/열등”, “위험한 사람”, “정상/비정상”처럼 사용자를 낙인찍는 표현을 쓰지 않습니다.

권장 문체:

- prompt: 35~75자 안팎의 한 문장. 문제 상황과 충돌하는 가치를 분명히 보여줍니다.
- choice: 35~80자 안팎. 선택의 명분과 감수해야 할 비용이 함께 느껴지게 씁니다.
- 어조: “먼저 둔다”, “제한한다”, “유지한다”, “좁게 예외를 둔다”, “열어둔다”, “고친다”처럼 사용자의 선택 행동이 보이는 동사를 사용합니다.
- 느슨한 상태 표현보다 책임 주체와 실제 행동을 드러냅니다. 예를 들어 `애매합니다`보다 `책임질 사람이 분명하지 않습니다`, `당번이 비어 있습니다`보다 `정리할 사람이 빠집니다`처럼 씁니다.
- 사물이나 개념이 사람처럼 행동하는 표현은 줄입니다. `같은 자리를 원합니다`보다 `같은 순서에 들어가려 합니다`처럼 실제 운영 장면의 행동으로 바꿉니다.
- 금지에 가까운 반복: 모든 Phase 2 선택지를 `찬성한다/반대한다`로 시작하는 방식.

리라이팅 예시:

| 구분 | 문구 |
|---|---|
| 피해야 할 prompt | 공동 안전을 이유로 개인 정보 일부를 임시로 모아야 하는 상황입니다. |
| 권장 prompt | 아파트 단톡방에서 분실 사고를 막자며 출입 기록 공유를 요구합니다. |
| 피해야 할 A | 찬성한다. 위험이 크다면 공동 안전을 위한 최소한의 정보 수집은 필요하다. |
| 권장 A | 불편해도 반복 피해를 줄이는 확인 절차를 먼저 둔다. |
| 피해야 할 B | 반대한다. 목적이 좋아도 개인의 통제권을 쉽게 넘겨서는 안 된다. |
| 권장 B | 불안해도 개인의 출입 기록은 함부로 공유하지 않는다. |

표시용 `display` 예시:

| 필드 | 값 |
|---|---|
| `display.promptLines` | `["아파트 단톡방에서 분실 사고를 막자며", "출입 기록 공유를 요구합니다."]` |
| `display.promptHighlights` | `["기록"]` |
| `display.choices.A.label` | `먼저 막기` |
| `display.choices.A.body` | `불편해도 반복 피해를 줄이는 확인 절차를 먼저 둔다.` |
| `display.choices.B.label` | `동의 먼저` |
| `display.choices.B.body` | `불안해도 개인의 출입 기록은 함부로 공유하지 않는다.` |

리라이팅 시에는 `id`, `axis`, `phase`, `branchCondition`, `scoreEffect`, `intensityEffect`를 바꾸지 않고 `prompt`, `choices`, `display`만 수정합니다.

#### Phase 1 문항

- 두 선택지가 같은 축의 양끝 가치를 직접 비교해야 합니다.
- 특정 정당, 정치인, 실제 선거명은 사용하지 않습니다.
- 한 선택지를 비상식적이거나 악하게 보이게 만들지 않습니다.
- 선택지 길이는 모바일 카드에서 읽기 쉽도록 35자~70자 안팎을 권장합니다.
- 사용자가 빠르게 이해해야 하므로, 너무 극단적인 사례보다 일상적이고 직관적인 가치 충돌을 우선합니다.
- V2/V3 variant도 같은 축의 추상 문장을 단어만 바꾸지 않고, 다른 생활 장면이나 다른 비용 구조로 다시 씁니다.
- variant 후보는 `가까운 판단 기준`, `처음 세운 기준`, `원칙을 계속`, `반대쪽 요구`, `쪽을 먼저 보존`처럼 fallback으로 생성된 일반 문구가 남지 않게 검수합니다.

#### Phase 2 한쪽 쏠림 분기 문항

- Phase 1에서 선택한 가치의 한계 비용을 드러내야 합니다.
- `principle` 선택지는 사용자가 기존 원칙을 계속 밀고 가는 답변이어야 합니다.
- `concession` 선택지는 반대 이념으로 전향하는 답변이 아니라, 예외와 한계를 인정하는 답변이어야 합니다.
- 한계 상황은 사용자가 앞서 고른 기준의 비용이 드러나는 딜레마여야 합니다. 예를 들어 확인 절차가 감시처럼 느껴지거나, 자유를 열어둔 결과 피해가 반복되는 식입니다.
- 단, 문항이 공포를 과장하거나 사용자를 압박하는 방식으로 쓰이면 안 됩니다. “한계 비용”은 보여주되, 선택지는 차분하게 유지합니다.

#### Phase 2 BALANCED 문항

- 양쪽 가치가 정면충돌해야 하며, 어느 쪽도 명백한 정답처럼 보이면 안 됩니다.
- A/B는 각각 해당 축의 양쪽 지표에 `+20`을 부여합니다.
- `intensityEffect`는 양쪽 모두 `neutral`입니다.
- 목적은 최종 알파벳 방향 결정이지, 원칙 고수형 판별이 아닙니다.
- 양쪽 모두 장점과 손실이 있어야 합니다. “안전하지만 답답함”, “자유롭지만 위험 부담”처럼 비용이 같이 드러나는 문항이 좋습니다.
        

## 3. 문항 및 분기 트리 상세 설계

## 4. 최종 결과 산출 및 태그(S/M) 판별 공식

모든 20문항의 응답이 끝나면, 각 축의 득점 결과를 합산하여 최종 알파벳 4자리와 강도 태그를 부여합니다. MVP에서는 총 문항 수를 20문항으로 고정하며, 가변 문항 수는 후속 버전에서 별도 설계합니다.

### 4.1. 알파벳 판별 (승자독식)

- **C vs I 축:** C 점수 > I 점수 ➡ **C** 배정. I 점수 > C 점수 ➡ **I** 배정.
    
- **T vs P 축:** T 점수 > P 점수 ➡ **T** 배정
    
- **M vs E 축:** M 점수 > E 점수 ➡ **M** 배정
    
- **O vs L 축:** O 점수 > L 점수 ➡ **O** 배정
    
- _예시 결과: CTMO, IPEL 등 16가지 조합._

#### 동점 처리 규칙

축별 점수가 동점이면 무작위로 배정하지 않습니다. 결과 재현성과 통계 일관성을 위해 아래 순서로 deterministic하게 결정합니다.

1. 해당 축의 Phase 2 응답 중 마지막으로 점수를 획득한 지표를 우선합니다.
2. Phase 2에서도 양측 점수가 완전히 같으면 해당 축의 Phase 1 두 번째 문항에서 선택한 지표를 우선합니다.
3. 위 규칙으로도 결정할 수 없는 비정상 이력은 `validation_error`로 처리하고 결과를 산출하지 않습니다.
    

### 4.2. 강도 태그 (S / M) 판별 로직

알파벳이 결정된 후, 각 축에서 최종 승리한 지표가 Phase 2의 극한 딜레마에서 얼마나 자주 원칙을 고수했는지 계산하여 태그를 결정합니다. Phase 2의 '원칙 고수' 선택지는 서버 전용 문항 데이터의 `intensityEffect`가 `principle`인 선택지입니다.

- **조건식 1 (원칙 고수형, Strong):** 4개의 승리한 지표 중 **3개 이상이 Phase 2 원칙 고수 선택을 2회 이상** 기록한 경우.
    
    - _해석:_ Phase 1에서 한쪽을 택한 뒤, Phase 2의 한계 딜레마에서도 전혀 타협하지 않고 원칙을 고수한 확고한 이념형.
        
    - _출력:_ `CTMO-S` (공동체의 원칙 설계자 - 원칙 고수형)
        
- **조건식 2 (상황 조정형, Moderate):** 위 조건을 충족하지 않는 모든 경우.
    
    - _해석:_ 상황에 따라 타협할 줄 아는 유연성을 지녔거나, 양쪽의 가치를 모두 고려하는 상황 조정형.
        
    - _출력:_ `CTMO-M` (공동체의 원칙 설계자 - 상황 조정형)
        

### 4.3. 데이터 무결성 검증 (백엔드 처리)

- 클라이언트(브라우저)에서 계산된 점수를 그대로 신뢰하지 않습니다.
    
- 반드시 클라이언트가 전송한 `답변 History (예: Q_1_C_I_1: A, Q_2_C_1: B)` 배열을 바탕으로 서버에서 직접 가중치를 재합산하여 최종 결괏값을 도출하고 DB에 `INSERT` 해야 합니다. (악의적 조작 방지)

### 4.4. 서버 계산 의사코드

아래 의사코드는 `/api/next-question`과 `/api/submit-result`의 기준 동작입니다. 실제 구현은 TypeScript로 작성하되, 분기 순서와 계산 의미는 이 흐름을 따릅니다.

```ts
const AXIS_ORDER = ['C_I', 'T_P', 'M_E', 'O_L'];
const AXIS_SIDES = {
  C_I: ['C', 'I'],
  T_P: ['T', 'P'],
  M_E: ['M', 'E'],
  O_L: ['O', 'L']
};

function computeScores(history, questions) {
  const scores = { C: 0, I: 0, T: 0, P: 0, M: 0, E: 0, O: 0, L: 0 };

  for (const answer of history) {
    const question = questions.byId(answer.questionId);
    const effect = question.scoreEffect[answer.choice] ?? {};

    for (const [key, value] of Object.entries(effect)) {
      scores[key] += value;
    }
  }

  return scores;
}

function historyForAxis(history, axis, questions) {
  return history.filter((answer) => questions.byId(answer.questionId).axis === axis);
}

function branchForAxis(axis, phase1History, questions) {
  const [left, right] = AXIS_SIDES[axis];
  const axisScores = computeScores(historyForAxis(phase1History, axis, questions), questions);

  if (axisScores[left] === axisScores[right]) return 'BALANCED';
  return axisScores[left] > axisScores[right] ? left : right;
}

function winnerForAxis(axis, scores, history, questions) {
  const [left, right] = AXIS_SIDES[axis];

  if (scores[left] > scores[right]) return left;
  if (scores[right] > scores[left]) return right;

  const phase2AxisAnswers = historyForAxis(history, axis, questions)
    .filter((answer) => questions.byId(answer.questionId).phase === 2)
    .filter((answer) => {
      const effect = questions.byId(answer.questionId).scoreEffect[answer.choice] ?? {};
      return effect[left] || effect[right];
    });

  const lastScoredAnswer = phase2AxisAnswers.at(-1);
  if (lastScoredAnswer) {
    const effect = questions.byId(lastScoredAnswer.questionId).scoreEffect[lastScoredAnswer.choice] ?? {};
    if (effect[left]) return left;
    if (effect[right]) return right;
  }

  const secondPhase1Answer = historyForAxis(history.slice(0, 8), axis, questions).at(1);
  if (!secondPhase1Answer) throw validationError();

  const effect = questions.byId(secondPhase1Answer.questionId).scoreEffect[secondPhase1Answer.choice] ?? {};
  if (effect[left]) return left;
  if (effect[right]) return right;

  throw validationError();
}

function nextQuestion(history, questions) {
  validateReachableHistory(history, questions);

  if (history.length >= 20) {
    return { status: 'completed', nextAction: 'submit_result' };
  }

  const nextIndex = history.length + 1;

  if (nextIndex <= 8) {
    const axis = AXIS_ORDER[(nextIndex - 1) % 4];
    const commonOrder = nextIndex <= 4 ? 1 : 2;
    return questionBy({ phase: 1, axis, branchCondition: 'COMMON', commonOrder });
  }

  const phase2Index = nextIndex - 9;
  const axis = AXIS_ORDER[phase2Index % 4];
  const branchOrder = Math.floor(phase2Index / 4) + 1;
  const branchCondition = branchForAxis(axis, history.slice(0, 8), questions);

  return questionBy({ phase: 2, axis, branchCondition, branchOrder });
}

function resultCode(history, questions) {
  validateReachableHistory(history, questions);
  if (history.length !== 20) throw validationError();

  const scores = computeScores(history, questions);
  const letters = AXIS_ORDER.map((axis) => winnerForAxis(axis, scores, history, questions));
  const intensityTag = computeIntensityTag(letters, history, questions);

  return `${letters.join('')}-${intensityTag}`;
}

function computeIntensityTag(letters, history, questions) {
  let strongAxisCount = 0;

  for (const letter of letters) {
    const principleCount = history.filter((answer) => {
      const question = questions.byId(answer.questionId);
      return question.phase === 2
        && question.branchCondition === letter
        && question.intensityEffect[answer.choice] === 'principle';
    }).length;

    if (principleCount >= 2) strongAxisCount += 1;
  }

  return strongAxisCount >= 3 ? 'S' : 'M';
}
```

## 5. 서버 구현 검증 규칙

문항 풀을 배포하기 전에 서버 또는 테스트 코드에서 아래 조건을 검증해야 합니다.

### 5.1. 문항 풀 정합성

- 축은 `C_I`, `T_P`, `M_E`, `O_L` 네 개만 존재해야 합니다.
- 각 축에는 Phase 1 `COMMON` 문항이 `commonOrder=1`, `commonOrder=2`로 정확히 2개 있어야 합니다.
- 각 축에는 Phase 2 왼쪽 지표, 오른쪽 지표, `BALANCED` 분기별로 `branchOrder=1`, `2`, `3` 문항이 정확히 3개씩 있어야 합니다.
- MVP 최소 알고리즘 슬롯 수는 44개입니다. 이보다 적으면 서버 시작 또는 빌드 검증 단계에서 실패해야 합니다.
- `phase=1` 문항에는 `branchOrder`를 넣지 않고, `phase=2` 문항에는 `commonOrder`를 넣지 않습니다.
- `phase=1` 문항의 `branchCondition`은 항상 `COMMON`이어야 합니다.
- `phase=2` 문항의 `branchCondition`은 해당 축의 두 지표 중 하나 또는 `BALANCED`여야 합니다.

### 5.2. 점수 및 강도 필드 정합성

- 모든 선택지는 `A`와 `B`를 반드시 가져야 합니다.
- `scoreEffect`의 점수는 MVP에서 `20` 또는 빈 객체만 허용합니다.
- Phase 1 문항과 Phase 2 `BALANCED` 문항의 `intensityEffect`는 양쪽 모두 `neutral`이어야 합니다.
- Phase 2 한쪽 쏠림 분기 문항은 한 선택지가 `principle`, 다른 선택지가 `concession`이어야 합니다.
- `principle` 선택지는 해당 분기 지표에 `+20`을 부여해야 합니다.
- `concession` 선택지는 점수를 부여하지 않습니다. 반대 지표에 점수를 주지 않는 이유는 "한계 인정"을 반대 이념 전향이 아니라 강도 완화로 해석하기 때문입니다.

### 5.3. 응답 이력 검증

- `history.length`는 `/api/next-question`에서 0~20, `/api/submit-result`에서 정확히 20이어야 합니다.
- 같은 `questionId`가 두 번 이상 등장하면 `validation_error`로 처리합니다.
- 서버가 계산한 다음 문항 ID와 클라이언트가 제출한 다음 `history` 항목의 `questionId`가 일치하지 않으면 `validation_error`로 처리합니다.
- Phase 2 문항은 해당 축의 Phase 1 결과로 선택된 `branchCondition`과 일치해야 합니다.
- `answeredAt`은 선택 필드이므로 결과 계산에 사용하지 않습니다. 플레이 시간 검증은 별도 `playTimeSec` 값을 기준으로 합니다.

## 6. 테스트 Fixture 기준

구현 시 아래 fixture를 최소 단위 테스트 또는 통합 테스트 데이터로 준비합니다. fixture 이름은 권장값이며, 실제 파일명은 프로젝트 테스트 구조에 맞춰 조정할 수 있습니다.

| fixture | 목적 | 기대 결과 |
|---|---|---|
| `empty_history` | 첫 문항 반환 검증 | `/api/next-question`이 Phase 1 첫 문항 반환 |
| `phase1_common_complete` | Phase 1 8문항 완료 후 분기 계산 | 각 축의 `C/I/T/P/M/E/O/L/BALANCED` 조건 산출 |
| `all_left_strong` | 모든 축에서 왼쪽 지표와 principle 선택 | 예: `CTMO-S`처럼 `S` 태그 산출 |
| `all_left_moderate` | Phase 1은 왼쪽 지표, Phase 2에서 concession 다수 선택 | 예: `CTMO-M`처럼 `M` 태그 산출 |
| `balanced_axis` | 한 축의 Phase 1이 1:1 동점 | 해당 축 Phase 2에서 `BALANCED` 분기 선택 |
| `duplicate_question_id` | 같은 문항 ID 중복 제출 | `400 validation_error` |
| `unreachable_phase2` | 도달 불가능한 Phase 2 문항 제출 | `400 validation_error` |
| `too_short_history` | 19개 응답으로 결과 제출 | `400 validation_error` |
| `too_long_history` | 21개 응답 제출 | `400 validation_error` |
| `too_fast_play_time` | `playTimeSec < 20` | `429 abuse_suspected` |
| `all_type_codes_strong` | 16개 유형 전체 S 도달성 | `CTMO-S`부터 `IPEL-S`까지 모두 산출 |
| `all_type_codes_moderate` | 16개 유형 전체 M 도달성 | `CTMO-M`부터 `IPEL-M`까지 모두 산출 |

### 6.1. Fixture 작성 규칙

- fixture는 `history` 배열과 기대 `resultCode`, `intensityTag`, `nextQuestionId`를 함께 보유합니다.
- 각 fixture의 문항 ID는 실제 `questions.ko-KR.ts`에 존재해야 합니다.
- `answeredAt`은 결과 계산에 영향을 주지 않으므로 테스트에서는 생략 가능합니다.
- `playTimeSec` abuse 테스트를 제외한 정상 제출 fixture는 `playTimeSec >= 20`을 사용합니다.
- `scoreEffect`, `branchCondition`, `intensityEffect`, `metadata`가 `/api/next-question` 공개 응답 어디에도 노출되지 않는지 재귀 검증합니다.

### 6.2. 콘텐츠 QA 기준

문항 검증은 구조 검증과 콘텐츠 검증을 분리합니다.

| 구분 | 통과 기준 |
|---|---|
| 구조 QA | 44개 슬롯 존재, ID 고유성, 분기 도달성, 점수/강도 필드 정합성 |
| 알고리즘 QA | fixture가 기대 결과와 일치하고 invalid history가 차단됨 |
| 콘텐츠 QA | 민감 주제 표현, 정치 진영 낙인, 허위 사실 단정, 혐오/차별 표현이 없음 |

콘텐츠 QA에서 발견된 문항 표현 문제는 알고리즘 구조를 바꾸지 않고 `prompt`와 `choices`만 수정하는 것을 원칙으로 합니다.
